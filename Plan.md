Plan: ConsafeWorkflow.Mcp — Stateful Workflow MCP Server (Shell)
Problem & Approach
Build an empty shell of a C# MCP (Model Context Protocol) server named
ConsafeWorkflow.Mcp. It is designed to grow into a stateful workflow engine that
directs Copilot agents through component-creation workflows. For now it exposes a
single tool that returns a hardcoded placeholder, with the surrounding structure
(session state, plug-in points, IDE registration, example agent) in place and
designed for future extension.

Architecture (from the supplied flowchart):

Agent invokes the MCP tool with a sessionId + userInput.
Server looks up WorkflowSession by session id (in-memory dictionary).
NEW session → returns initialization message; established → advances state machine
and returns the next directed instruction.
Tool output is a directed message the agent presents verbatim to the user.
This is the shell only: the lookup/advance logic returns a hardcoded string today,
but the seams (session model, state machine hook, local-model hook) are stubbed and
documented.

Tech choices
.NET 10 (SDK 10.0.300 is installed) — net10.0 target.
Official ModelContextProtocol C# SDK + Microsoft.Extensions.Hosting.
stdio transport (matches .vscode/mcp.json "type": "stdio").

Design for future extensibility (stub now, not implemented)
IWorkflowEngine is where the real state machine goes; StubWorkflowEngine
currently returns the placeholder for all inputs.
ILocalModelClient is the documented plug-in point for a local OpenAI-compatible
endpoint (Ollama / LM Studio) — interface only, no implementation.
WorkflowSession is a plain serializable POCO (System.Text.Json friendly) so it
can later be persisted; ISessionStore abstracts storage to swap the in-memory
dictionary for a durable store.
Tool contract (shell)
get_next_step(sessionId: string, userInput: string):
Looks up/creates session in the store (so the seam is real).
Calls the engine, which returns "MCP shell — not yet implemented" for all inputs.
IDE integration
.vscode/mcp.json registers consafeworkflow as a stdio server using a relative
project path and --no-build. Works in both Visual Studio and VS Code.
README documents the required one-time dotnet build.
STATUS (2026-06-15) — RESOLVED: elicitation is now the primary input channel
Root cause confirmed: MCP tool arguments are generated by the agent/LLM, not piped
from the chat box, so userInput is unreliable (often empty / IDE context). The reliable
channel is elicitation — the server shows its own textbox and gets the typed value
back. Pivot implemented: elicitation is now the PRIMARY input path, not a fallback.

Key fixes
Visual Studio sends action="accepted" (not the spec's "accept") and leaves
IsAccepted=false. Parser now matches any accept* (case-insensitive) or IsAccepted.
Confirmed working: VS renders the dialog and returns the typed value.
Tolerant content parsing: prefer reply, else first usable string property; coerce
primitives via raw text. Added a [elicit] raw response: stderr diagnostic.
Added IWorkflowEngine.InitialPrompt so a brand-new session has an opening question to
elicit on the very first call (fixes the old "PendingPrompt null → skip elicit").
Tool flow now: question = PendingPrompt ?? engine.InitialPrompt; if elicitation
available → elicit(question) → advance → return next (primary); else two-phase fallback
(return question, then advance with userInput next turn).
${workspaceFolder} in .mcp.json / .vscode/mcp.json fixed the cwd startup error.
Seam noted in WorkflowTool: elicited answer is where ILocalModelClient would route input
to a local model (Ollama/LM Studio) to normalise/suggest values.
